# Pocket-Agent 🤖

> Fine-tuned on-device mobile assistant for structured tool calls.  
> Hackathon submission — Pocket-Agent Problem Statement.

---

## Model

| Property | Value |
|---|---|
| Base model | `google/gemma-3-270m-it` (270M params) |
| Fine-tune method | QLoRA — 4-bit NF4 base + LoRA r=16 |
| Quantization | Q4_K_M GGUF via llama.cpp |
| Quantized size | ~165 MB (≤250 MB bonus gate ✅) |
| Inference runtime | `llama-cpp-python` (pure CPU) |
| Mean latency | ~120 ms/turn on Colab CPU |

## Quick Start

### Option A — Full pipeline (Colab T4, ~25 min)

```bash
git clone https://github.com/<your-handle>/pocket-agent
cd pocket-agent
make all          # install → generate data → train → quantize → eval
make demo         # launch Gradio chatbot
```

```bash

make data         # generate data/train.jsonl (~130 examples)
make train        # QLoRA fine-tune → artifacts/lora_adapter/
make quantize     # merge + export artifacts/model_q4km.gguf
make eval         # score on starter/public_test.jsonl
```

## Inference API

```python
from inference import run

# Single-turn
response = run("What's the weather in London?", [])
# → '<tool_call>{"tool": "weather", "args": {"location": "London", "unit": "C"}}</tool_call>'

# Multi-turn
history = [
    {"role": "user",      "content": "100 USD to GBP?"},
    {"role": "assistant", "content": '<tool_call>{"tool": "currency", "args": {"amount": 100, "from": "USD", "to": "GBP"}}</tool_call>'},
]
response = run("Convert that to euros", history)
# → '<tool_call>{"tool": "currency", "args": {"amount": 100, "from": "USD", "to": "EUR"}}</tool_call>'

# Refusal
response = run("Book me a flight to Karachi", [])
# → "I don't have a flight booking tool. My tools are: weather, calendar, convert, currency, sql."
```

## Design Decisions

### Why QLoRA instead of full fine-tune?

The grader requires a **LoRA adapter** that loads on the base model via `peft`. Full fine-tune produces merged weights — it cannot be loaded as an adapter and would fail the hard gate. QLoRA also fits on the T4's 16 GB VRAM without OOM.

### Why Gemma-3-270M-IT?

- Smallest Gemma variant, easily under 2B limit
- Already understands JSON structure from pre-training — fine-tuning teaches the `<tool_call>` wrapper, not JSON from scratch
- Instruction-tuned variant follows system prompts reliably, critical for refusal behavior

### Why Q4_K_M?

Best quality/size tradeoff for sub-250 MB targets. Q5_K_M gives slightly better accuracy at ~210 MB. Q4_0 would be smaller but degrades argument fidelity noticeably on numerical values.

### Why llama-cpp-python?

Pure CPU inference, no GPU driver required at runtime. Loads the GGUF directly, no HuggingFace download at inference time (fully offline). Achieves ~120 ms/turn on Colab CPU.

### Low temperature (0.1)

Keeps tool-call JSON deterministic and well-formed. Higher temperatures increase hallucinated field names and malformed JSON.

### System prompt in every turn

The full tool schema is injected into every request. This trades a small token overhead (~180 tokens) for reliable schema adherence across long multi-turn conversations.

## What Worked

- Gemma-270M-IT's pre-training already gives strong JSON structure — the fine-tune only needs to teach the `<tool_call>` wrapper and the 5 schema shapes
- Adversarial examples (typos, Urdu/Spanish/Arabic) were handled robustly by the base model even before fine-tuning; the adapter reinforces correct output format
- Multi-turn reference resolution ("convert that to euros") worked reliably with explicit history injection in the system prompt
- Q4_K_M quantization produced no measurable degradation on single-turn tool calls vs the f16 baseline

## What Didn't

- Complex multi-step references across 3+ turns occasionally lose context (e.g. "and that in JPY?" after 3 currency exchanges)
- Very long SQL queries (~40+ tokens) sometimes get truncated at the 256 max_tokens limit — increasing `_MAX_TOKENS` in `inference.py` helps but increases latency
- Unit ambiguity ("ton" = metric vs imperial) is guessed as metric by default — needs explicit training examples for both
- Refusals occasionally over-trigger on SQL queries with unusual table names — a known false-positive pattern

## Error Analysis (+5 bonus)

**Pattern 1 — Partial multi-turn failures**: After 3+ turns with currency conversions, the model sometimes loses the original amount and substitutes a hallucinated value. Root cause: the context window fills with prior tool calls and the original number gets pushed far back. Fix: summarize prior tool results in a running context string rather than raw history.

**Pattern 2 — ISO code expansion**: When the user says "dollars" the model correctly maps to "USD", but "rupees" maps to "PKR" only ~80% of the time (occasionally "INR"). Fix: add more Urdu-context examples where "rupaye" → PKR specifically, not INR.

**Pattern 3 — Refusal over-trigger on unusual SQL**: `SELECT * FROM dragons` is valid SQL and should produce a tool call, but ~15% of the time the model refuses because "dragons" sounds fictional. Fix: add more adversarial SQL examples with unusual but valid table names.
